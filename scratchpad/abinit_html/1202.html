30     class copier:
31         "Smart-copier (YAPTU) class"
32         def copyblock(self, i=0, last=None):
33             "Main copy method: process lines [i,last) of block"
34             def repl(match, self=self):
35                 "return the eval of a found expression, for replacement"
36                 # uncomment for debug: print '!!! replacing',match.group(1)
37                 expr = self.preproc(match.group(1), 'eval')
38                 try: return str(eval(expr, self.globals, self.locals))
39                 except: return str(self.handle(expr))
40             block = self.locals['_bl']
41             if last is None: last = len(block)
42             while i<last:
43                 line = block[i]
44                 match = self.restat.match(line)
45                 if match:   # a statement starts "here" (at line block[i])
46                     # i is the last line to _not_ process
47                     stat = match.string[match.end(0):].strip()
48                     j=i+1   # look for 'finish' from here onwards
49                     nest=1  # count nesting levels of statements
50                     while j<last:
51                         line = block[j]
52                         # first look for nested statements or 'finish' lines
53                         if self.restend.match(line):    # found a statement-end
54                             nest = nest - 1     # update (decrease) nesting
55                             if nest==0: break   # j is first line to _not_ process
56                         elif self.restat.match(line):   # found a nested statement
57                             nest = nest + 1     # update (increase) nesting
58                         elif nest==1:   # look for continuation only at this nesting
59                             match = self.recont.match(line)
60                             if match:                   # found a contin.-statement
61                                 nestat = match.string[match.end(0):].strip()
62                                 stat = '%s _cb(%s,%s)\n%s' % (stat,i+1,j,nestat)
63                                 i=j     # again, i is the last line to _not_ process
64                         j=j+1
65                     stat = self.preproc(stat, 'exec')
66                     stat = '%s _cb(%s,%s)' % (stat,i+1,j)
67                     # for debugging, uncomment...: print "-> Executing: {"+stat+"}"
68                     exec_(stat, self.globals, self.locals)
69                     i=j+1
70                 else:       # normal line, just copy with substitution
71                     self.ouf.write(self.regex.sub(repl,line))
72                     i=i+1
73         def __init__(self, regex=_never, dict={},
74                 restat=_never, restend=_never, recont=_never,
75                 preproc=identity, handle=nohandle, ouf=sys.stdout):
76             "Initialize self's attributes"
77             self.regex   = regex
78             self.globals = dict
79             self.locals  = { '_cb':self.copyblock }
80             self.restat  = restat
81             self.restend = restend
82             self.recont  = recont
83             self.preproc = preproc
84             self.handle  = handle
85             self.ouf     = ouf
86         def copy(self, block=None, inf=sys.stdin):
87             "Entry point: copy-with-processing a file, or a block of lines"
88             if block is None: block = inf.readlines()
89             self.locals['_bl'] = block
90             self.copyblock()

122     __doc__ = """
123     We may often want to copy some "template" text (normally from an input file) to an output
124     file-like object, while expanding Python expressions (and possibily executing Python
125     statements, e.g. for selection or repetition) that may be "embedded" in the template text.
126     
127     YAPTU is a small but complete Python module for this purpose, suitable for processing most
128     any kind of structured-text input, since it lets client-code decide which
129     regular-expressions will denote embedded Python expressions and/or statements (so, such
130     re's can be selected to avoid conflicting with whatever syntax is needed by the kind of
131     structured-text that is being processed -- be it HTML, a programming language, RTF, ...).
132     
133     The compiled-re object that identifies expressions, if not None, is used for a .sub on
134     each line of the input; for each MatchObject "match" that results, match.group(1) is
135     eval'd as a Python expression, and the result, transformed to a string, gets substituted
136     in place; a dictionary to be used as the namespace for the evaluation must also be passed
137     as an argument. Many such (non-overlapping!) matches per line are possible, but the
138     resulting text is NOT re-scanned for 'embeddings'.
139     
140     Python statements can also be embedded; this is mostly intended to be used with
141     if/elif/else, for, while, and is line-based. Statement-related lines are recognized
142     through three more regular-expression objects that are passed in, one each for
143     'statement', 'continuation', 'finish', to be used for regular-expression _match_ (i.e.,
144     from line start) [again, each can be None if no such statements are to be embedded].
145     
146     The 'stat' and 'cont' re's are followed by the corresponding statement lines (beginning
147     statement, and continuation statement, respectively -- the latter normally makes sense
148     only if it's an 'else' or 'elif'). Statements can nest without limits.
149     
150     If a statement must be embedded that does NOT end with a colon (e.g., an assignment
151     statement), then a Python comment MUST terminate its line; conversely, such comments are
152     NOT allowed on the kind of statements most often embedded (if, else, for, while) --
153     _their_ lines must terminate with their ':' (optionally followed by whitespace). This
154     peculiarity is due to the somewhat tricky technique used in YAPTU's implementation:
155     embedded statements (with their continuations) are exec'd with _recursive calls to yaptu's
156     copyblock function_ substituted in place of the blocks of template-text they contain,
157     taking advantage of the fact that such a single-statement "suite" can be placed on the
158     same line as the controlling statement, right after the colon, and this avoids any
159     whitespace-issue (yaptu does NOT rely on whitespace to discern embedded-statement
160     structure, but on the explicit statement/continuation/end markers!).
161     
162     Net of comments, whitespace, and docstrings, YAPTU is just 50 source-lines of code, but
163     rather a lot happens within that small compass. Instances of the _nevermatch auxiliary
164     class are used in lieu of regular-expression objects that are passed in as 'None' -- their
165     polymorphism with compiled-re objects (regarding the only two methods of the latter that
166     yaptu uses, .sub and .match) saves quite a few tests in the main body of code, and
167     simplifies it -- a good general idiom to keep in mind.
168     
169     An instance of the 'copier' class has a certain amount of state, besides the relevant
170     compiled-re's (or nevermatch instances) and the output file-like object being used (the
171     latter need only implement method .write), that is held in two dictionary attributes --
172     self.globals, the dictionary that was originally passed in for expression-substitution,
173     and self.locals, another dictionary which is used as the local-namespace for all of
174     yaptu's exec and eval uses. Two internal-use-only items in self.locals, in particular
175     (with names starting with _) indicate the block of template-text being 'copied' (a
176     sequence of lines, each ending in a '\n'), at key '_bl', and the bound-method that
177     performs the copying, self.copyblock, at key '_cb'.
178     
179     Holding these two pieces of state in self.locals items is not quaint personal usage --
180     it's part of the key to yaptu's workings, since self.locals is what is guaranteed to be
181     made available to the code that yaptu exec's (self.globals, too, but yaptu does NOT dirty
182     THAT dictionary -- it is owned by its caller!). Since .copyblock must be recursive (the
183     simplest way to ensure no nesting limitations), it is important that nested recursive
184     calls be always able to further recurse, if needed, through their exec statements! Access
185     to _bl is similarly necessary -- .copyblock only takes as arguments the line _indices_
186     inside _bl that a given recursive call is processing (in the usual form -- index of first
187     line to process, index of first following line to AVOID processing; i.e.,
188     lower-bound-included, upper-bound-excluded, as everywhere in Python).
189     
190     copyblock's 32 SLOCs are the heart of YAPTU. The repl nested function is the one that is
191     passed to the .sub method of compiled RE objects to get the text to be used for each
192     expression substitution -- it uses eval on the expression string, and str() on the result
193     to ensure it, too, is turned back into a string. Most of copyblock is a simple while loop
194     that examines relevant block lines from the start; when it doesn't match a statement-start
195     RE, it copies the line out to the output file-object, with substitutions; when it does
196     match statement-start, it enters a smaller nested-loop looking for statement continuations
197     and statement-end (with proper accounting for nesting-levels, of course!). As it goes, it
198     builds up in local variable 'stat' a string, with the original statement [and possibly its
199     continuations at the same nesting-level] followed by a recursive call to _cb(i,j) right
200     after each semicolon [with newlines as separators between continuations, if any]. Lastly,
201     'stat' gets passed to the exec statement; the nested loop terminates, and the main loop
202     resumes from right after the embedded-statement just processed. Note that the
203     exec-statement will inevitably invoke copyblock recursively, but that does not disturb the
204     loops' state [based on local variables unoriginally named i and j, since they are
205     loop-counters and indices on the _bl list...], thanks to perfectly normal
206     recursive-invocation mechanisms.
207     
208     Are the slightly-tricky subtleties in yaptu justified -- or does it violate the Prime
209     Directive, to "do the simplest thing that can possibly work"? I lean towards the former
210     opinion -- that there is no _gratuitous_ subtlety in yaptu, that it uses the minimal
211     amount of trickiness compatible with doing its job sensibly -- its job being to supply a
212     reusable templating utility, flexible, effective, and small, for a variety of actual and
213     potential uses; I can be biased, but I can see no _substantial_ simplification that could
214     be made for the sake of clarity without impairing functionality. However, your comments
215     (and proposed rewrites!) are welcome, of course!
216     
217     Late-breaking additions: all expressions and statements may now be "preprocessed" by
218     passing an optional callable "preproc" when creating the copier -- default is no
219     preprocessing. Exceptions in expressions (only) may be handled by passing an optional
220     callable "handle" (default is re-raising the exception, which terminates YAPTU's
221     processing and propagates outwards).
222     """
